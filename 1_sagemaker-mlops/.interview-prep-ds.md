## Foundational ML/DS Questions

### 1. Boosting vs Bagging: What's the Difference?
Bagging and boosting both use ensemble methods, but with different strategies. Bagging trains multiple models in parallel on bootstrap samples and averages their predictions. It reduces variance by averaging out noise, making it great for high-variance models like deep decision trees.

Boosting trains models sequentially, where each model focuses on samples misclassified by the previous model. This reduces bias by forcing the ensemble to fit harder on difficult cases.

When to choose:

Use bagging (Random Forest) when overfitting is your problem (complex model, noisy data)
Use boosting (XGBoost) when underfitting is your problem (model's too simple) or you need feature importance
The key tradeoff: Bagging reduces variance, boosting reduces bias.

### 2. Overfitting: Detection and Prevention
How do you detect overfitting in a model?
There are many ways to detect overfitting in a model. One of my favorite ways is to set aside data for a holdout or test set and evaluate the performance of the model on this unseen data. It is important to ensure that there isn't any information leakage between the data used to train and this test data, because it will not be indicative of what is available at inference. We then make predictions on both the train and test data and compare the aggregated optimization metric over the two data sets. If the metric is roughly the same for both, then there is little evidence to suggest overfitting. If the metric performs well over training but poorly over testing then there is strong evidence to suggest that we are overfitting to the training data.

What techniques would you use to prevent it? 
We can use regularization to penalize model complexity. If we are using a tree based model that we can decrease max_depth or decrease min_child_weight. We can implement early stopping if our performance metric isn't meaningfully decreasing at each new epoch. We can rely on cross validation, which is a more robust version of a holdout set. in k-fold cross validation we train over all of the data, but we split the data into K folds. For each fold, we train on K-1 folds and validate on the holdout fold. This repeats K times, using all data for training but never training on the same data we validate on.

What's the relationship between model complexity, data size, and overfitting risk?
If a model is highly complex (for example has a lot of hyperparameters to tune) then the risk of overfitting increases. This is because there are fewer degrees of freedom left over our data where hyperparameters are set aside to be optimized for during tuning. Similarly, the less data we have the more likely we are to overfit as well. Again, this is a degrees of freedom issue. With fewer repetitions, it is harder to say definitively whether the relationship that a model has learned is true or fitted to noise.

### 3. Feature Scaling: Why and When?
When is feature scaling necessary? 
Feature scaling can be useful when features take on different ranges and distributions from one another. It can be shown that a disproportionate amount of weight will be given to features with higher variance, so it is often useful to scale features to the same variance or range when possible. Algorithms like KNN, K-means, SVM, sometimes linear/logistic regression need scaling. Decesion trees don't need scaling because the splits are scale-invariant.

What's the difference between standardization and normalization?
Standardization rescales features so that the data has a mean of zero and a standard deviation of 1 while normalization rescales the range of the data to be bounded within [0,1].

Would you scale features before or after train/test split, and why?
I would scale feature after train/test split. I would take the parameters used for scaling for the training data and apply those parameters to then scale the test split. I would do this because it is most indicative of how rescaling will need to be applied at inference. At inference we don't get the advantage of knowing the full rescale parameters and are inherently limited by the historical parameters, so I would want my testing data to replicate the same generating process as production so that I can most accurately assess the performance metric tradeoff of scaling the data.

### 4. Cross-Validation Strategy
Explain k-fold cross-validation.
k-fold cross-validation is where we break the data into k folds and then train a model on every combination of k-1 folds, using the holdout fold for validation. We train k models and report the average validation score across the k folds. For production, we typically retrain once on all data using the validated hyperparameters. (We don't actually use all k models in production; we use CV only for evaluation.)

When would you use stratified k-fold vs regular k-fold?
Use stratified k-fold when you have imbalanced classes. If 95% of your data is negative and 5% positive, regular k-fold might randomly create a fold with only negative examples. That fold's validation score would be meaningless (just predicting all negative). Stratified k-fold ensures each fold maintains the 95/5 distribution.

What are the pitfalls of using cross-validation incorrectly on time-series data?
using regular cross-validation on time-series data is always a bad idea, because the validation split will never match the data generating process at inference. The data we get at inference is always limited to what we know at time T, so all we will ever have is information at time T - s (s some arbitrary time segment). If we use a regular cross-validation then we will mix in our knowledge of the past, present, and future to predict some time T, an advantage that we will never have in production. It is always better to use a time-series cross-validation, where the data is split into k folds defined by T-k, T-k+1, T-k+2, ... This will not evenly distribute the data unless we segment the duration to a fixed amount of time as opposed to using all available history.


### 5. Imbalanced Classification
You have a dataset where the positive class is 2% of data. How would you handle this?
First, I'd clarify the business goal: Do we care more about precision (few false positives) or recall (catch all positives)? Then:

Cost-weighted loss (my preferred approach)
Penalizes misclassifying the minority class during training
Empirically tune the cost weight (1, 5, 10, 50) via cross-validation
No data loss, no artificial data
SMOTE (if cost-weighting isn't enough)
Generates synthetic positive examples in feature space
More data for model to learn from
Can combine with undersampling for balance
Threshold adjustment (as fallback)
Lower prediction threshold to catch more positives
But sacrifices precision (more false positives)
Only use if business can tolerate more FP
Why NOT resampling alone: Resampling creates artificial distributions that don't reflect production data. Better to resample for training but evaluate on real distribution.

What are pros/cons of resampling vs cost-weighted loss vs threshold adjustment?
resampling is nice because you can show your model where you want it to learn the discrepancy, but it isn't representative of the kind of data that you will see in production. In production we don't get to throw rows for inference at our model that are convenient to the multi-dimensional regions that we showed our model. Cost-weighting directly addresses imbalance in the loss function. You can empirically determine the best cost weight through grid search or cross-validation. For example, try cost_weight = [1, 5, 10, 50] and pick the one with best F1 on validation data. Finally, threshold adjustment is okay too, but usually comes at the cost of high false positives so Precision might take a hit(in this context). 

### 6. Interpretability vs Performance Tradeoff
You have a complex deep learning model with 95% accuracy vs a linear model with 88% accuracy. 
How do you decide which to deploy? 
There are several factors that would determine whether to use the DNN or the OLS model.
1 Is model interpretability important? Linear models are extremely easy interpret because they assume that all of the relationships between the dependent and independent variable are linear. This assumption gives us a single coefficient per feature that explains the slope/relationship between that variable and the dependent variable. Further, we can also compute t statistics and p-values which test the hypothesis of whether the coefficient value is statistically significant (i.e. there is reasonable evidence to suggest that it did not come from a distribution where the population coefficient parameter is zero, which would mean that there is no relationship.). If model interpretability is not important, then the DNN will have the edge because of it's superior accuracy measurement. Given the discrepancy, the DNN has found some important non-linear relationships or interractions between the independent and dependent variables. Unfortunately, these relationships are notoriously difficult to explain if they can be explained at all.
2 Is training cost a factor? DNN models are usually more expensive to train then OLS models. If we want to cut the cost of computation and memory at the expense of our performance metric, then we would consider the switch from DNN to OLS.
3 Is the performance stable between traning iterations from the DNN to the OLS. The DNN will be more sensistive to overfitting. Methods like backtesting are useful to ensure the consistency in performance between trains as new data accumulates over time. OLS is usually less prone to overfitting because it makes harder assumptions of the relationships between the dependent and independent variables from the beginning.
If the model doesn't need to be interpreted, we have plenty of expenses for higher performance, and the performance is stable over backtesting then I would choose the DNN. If the model needs to be interpreted, we are low on costs, and other methods are not yielding consistent performance metrics across time then I would choose the OLS.

What changes that decision?
Regulatory requirement → Interpretability wins (no negotiation)
Deployment frequency (daily retraining) → Simpler model wins
Failure cost (what if wrong?) → Interpretability wins (easier to debug)
Data volume (10M rows vs 1M) → DNN more feasible
My default: Linear model unless accuracy gap is >5% AND business can tolerate failures. Then consider DNN or hybrid (LIME/SHAP).

### 7. P-value and Statistical Significance
Explain p-values. 

P(result | null is true) ← What p-value is
P(null is true | result) ← What people think it is

The p-value is the probability of observing your result (or more extreme) IF the null hypothesis were true. It's NOT the probability your result is wrong or the null is true. It's a conditional probability: P(data | null is true).

Example: If null=0 and you get a t-stat of -1 on a standard normal, p ≈ 0.32 (two-tailed).

What does p<0.05 actually mean? 
0.05 is an arbitrary threshold we choose (convention, not law). If we set α=0.05 and p<0.05, we reject the null. This means: if the null were true, we'd see results this extreme ≤5% of the time. We're willing to accept a 5% false positive rate.

Important: p<0.05 does NOT mean 5% chance we're wrong. It means, in the long run, if we repeat the experiment many times with the same α=0.05, ~5% of rejections will be false positives.

What's the multiple comparisons problem? 
If you run 20 independent tests at α=0.05, the family-wise error rate ≈ 1-(0.95^20) ≈ 64%. To fix this:

Bonferroni: α/number of tests (0.05/20 = 0.0025 per test)
Benjamini-Hochberg: Controls false discovery rate instead (less conservative)

Why is statistical significance different from practical significance?
Statistical: Does p<0.05? (arbitrary threshold)
Practical: Is the effect size large enough to matter in the real world?
Example: Drug reduces blood pressure by 0.5mmHg (p<0.001, statistically sig, but patient won't notice). Another reduces it by 20mmHg (p=0.06, not statistically sig, but patient feels better). Need BOTH considerations.

### 8. Correlation vs Causation
You observe that ice cream sales correlate with drowning deaths.
Should you reduce ice cream production?

Not necessarily. Correlation does not imply causation. There are many variables that are correlated that don't have any causal relationship to one another. There may be a latent variable that causes the correlation for both. For example, it is possible that warm weather drives up the demand for ice cream and also the demand for swimming which increases the chances of an individual drowning. Ice cream sales and drowning now become correlated dispite the fact that neither one were the cause.

How do you distinguish correlation from causation?

Gold standard (when possible): Randomized Controlled Trial
Randomize the exposure (e.g., offer ice cream vs not) and measure the outcome. Random assignment breaks all confounding paths because treatment is independent of baseline characteristics.

Real-world observational methods (when RCTs impossible/unethical):

Natural experiments — Exploit variation you didn't create (e.g., weather variation across regions)
Propensity score matching — Match treated and untreated units on confounders
Instrumental variables — Find a variable that affects exposure but NOT outcome (except through exposure)
Regression discontinuity — Exploit sharp cutoffs (e.g., policy thresholds)
Difference-in-differences — Compare groups before/after treatment
Causal frameworks:
I use DAGs (Directed Acyclic Graphs) to visualize confounding paths and apply the backdoor criterion: does a variable block all non-causal paths from treatment to outcome?

In the ice cream example:

Weather → Ice cream (confound path)
Weather → Drowning (confound path)
So weather is a confounder. Control for it (via matching, regression, stratification) and the association disappears.
My experience: At Corewell Health, I used causal inference to separate real impact of medical interventions from confounding factors like patient severity and baseline health.

### 9. Neural Networks Basics
Explain how backpropagation works.
Backpropagation takes the cost function of the final layer and uses its information of the gradients in each of the neurons for the previous layer to make adjustments to the weights and biases in proportion to their effect on minimizing the final cost. It will do this for all neurons in all previous layers until it has run out of layers. Backpropagation computes gradients using the chain rule, working backward from the output loss through each layer. Each weight update is proportional to ∂Loss/∂weight.

Why do we need activation functions? 
Without non-linearity, stacking linear transformations is just one big linear function (W₁ × W₂ = W₃). Activation functions introduce non-linearity, enabling the network to learn complex patterns. An activation function is a mathematical function applied to the weighted sum of inputs and bias at a neural network node to determine its output.  It serves as a decision gate that introduces non-linearity into the model, which is essential for enabling neural networks to learn complex, non-linear patterns and relationships in data rather than behaving as simple linear regression models.

What's the difference between a ReLU, sigmoid, and tanh activation? 
Sigmoid: Squashes inputs to a range of (0, 1).  It is not zero-centered, meaning its output is always positive.  This can cause gradients to move in "zig-zag" patterns during optimization, slowing down convergence.
Tanh (Hyperbolic Tangent): Squashes inputs to a range of (-1, 1).  Crucially, it is zero-centered, which generally makes it easier for the network to learn compared to Sigmoid, as the mean of the activations is closer to zero. 
ReLU (Rectified Linear Unit): Outputs 0 for negative inputs and the input itself for positive values, resulting in a range of [0, ∞).  It is not zero-centered and produces sparse activations (many neurons output exactly zero), which can improve computational efficiency and reduce overfitting. 

When would you use each?
Sigmoid: Highly susceptible to the vanishing gradient problem.  For very large or small inputs, the gradient becomes nearly zero, preventing weights in deep networks from updating effectively.
Tanh: Also suffers from the vanishing gradient problem at extreme values, but the gradient is generally stronger than Sigmoid, making it a better choice for hidden layers if a bounded output is required. 
ReLU: Effectively solves the vanishing gradient problem for positive inputs because its gradient is a constant 1.  This allows gradients to flow freely through deep networks, enabling faster training and convergence. However, it can suffer from the "Dying ReLU" problem, where neurons get stuck outputting zero if they receive only negative inputs. 

Rule of Thumb
Hidden Layers: Always start with ReLU. 
Output Layer:
Probability (0–1)? → Sigmoid
Any positive number? → ReLU or Linear
Any number? → Linear
Between -1 and 1? → Tanh 

### 10. Vanishing and Exploding Gradients
What are vanishing and exploding gradients? 
Vanishing gradients occur when gradients become extremely small during backpropagation in deep neural networks, causing weight updates in earlier layers to be negligible and stalling learning.  Exploding gradients happen when gradients grow excessively large, leading to unstable weight updates, numerical overflow (NaN values), and model divergence. 

Vanishing gradients: Gradients → 0 during backprop, early layer weights don't update, learning stalls
Exploding gradients: Gradients → ∞, weights diverge, NaN values, model crashes

Why do they happen in deep neural networks? 
These issues arise primarily due to the repeated multiplication of small or large values (such as weights or derivatives from activation functions like sigmoid or tanh) as gradients propagate backward through many layers.

What techniques would you use to prevent them? 
ReLU activation: Constant gradient=1 for positive inputs, no saturation
Xavier/He initialization: Scale weights by √(fan_in) or √(2/fan_in) to keep gradient magnitude ≈ 1 per layer
Gradient clipping: Cap ||gradient|| ≤ threshold (e.g., 5), prevents explosions
Batch Normalization: Normalize activations to mean=0, var=1, prevents saturation, applies learnable γ,β
Skip connections (ResNets): y = F(x) + x lets gradients bypass deep layers via shortcuts—KEY for 100+ layer networks

How does batch normalization help?
During training, the distribution of activations changes as previous layers update their weights. Batch normalization addresses this by normalizing the inputs of each layer for every mini-batch to have a mean of 0 and a variance of 1.  It then applies learnable scale ($\gamma$) and shift ($\beta$) parameters to preserve the network's representational power. 

By keeping activations within a stable range, batch normalization prevents them from drifting into the saturated regions of activation functions (like sigmoid or tanh) where derivatives are near zero.  This ensures that gradients remain large enough to flow effectively backward through the network, preventing the vanishing gradient problem. Simultaneously, it prevents activations from becoming excessively large, which helps avoid exploding gradients.

### 11. K-Means and Clustering Fundamentals
Explain how K-means clustering works. 
It is an iterative unsupervised learning algorithm that partitions unlabeled data into k distinct clusters by minimizing the distance between data points and their cluster centers. 

The process operates in three main phases:

Initialization: The algorithm begins by randomly selecting k data points to serve as initial centroids, or cluster centers. 
Assignment: Each data point in the dataset is assigned to the cluster whose centroid is nearest, typically using Euclidean distance. 
Update: The centroids are recalculated as the mean (average) of all data points currently assigned to that cluster. 
This assignment and update cycle repeats iteratively until the centroids stabilize (converge) or a maximum number of iterations is reached, resulting in clusters where points within a group are more similar to each other than to points in other groups.

How do you choose the number of clusters?
1. The Elbow Method
This is the most common visual approach. It involves plotting the Within-Cluster Sum of Squares (WCSS) or inertia against the number of clusters. 

How it works: As k increases, WCSS decreases because points are closer to their centroids. The goal is to find the "elbow" point where the rate of decrease sharply slows down. 
Selection: The optimal k is at this inflection point, representing a trade-off between minimizing error and model complexity. Beyond this point, adding more clusters yields diminishing returns.

2. Silhouette Analysis
While the Elbow method focuses on variance, the Silhouette Score measures how well-separated the clusters are. 

Metric: It calculates a score between -1 and 1 for each data point based on two factors:
Cohesion ($a$): The average distance to other points in the same cluster.
Separation ($b$): The average distance to points in the nearest neighboring cluster. 
Formula: $s = \frac{b - a}{\max(a, b)}$. 
Selection: You run K-means for a range of $k$ values and choose the one that yields the highest average silhouette score.  A score near 1 indicates dense, well-separated clusters, while a score near 0 or negative suggests overlapping or incorrect assignments.

What are the limitations of K-means? 
1. Sensitivity to Initialization
The algorithm is highly dependent on the random initial placement of centroids.  Different starting points can lead to convergence at different local optima rather than the global optimum, producing inconsistent results across multiple runs.
2. Requirement to Pre-specify K
Users must define the number of clusters (k) before running the algorithm.  In most practical scenarios, the true number of clusters is unknown, and choosing an incorrect $k$ leads to misleading partitions. Heuristic methods (Elbow, Silhouette) provide guidance but rarely yield unambiguous answers.
3. Assumption of Spherical Clusters
K-means assumes clusters are spherical (or hyperspherical) and isotropic (equal variance in all directions).  It fails to identify elongated, curved, or irregularly shaped clusters (e.g., crescents or rings), often splitting natural non-spherical groups or merging distinct ones to minimize variance.

When would you use hierarchical clustering instead?
1. Unknown Number of Clusters
Hierarchical clustering is the preferred choice when you do not know the optimal number of clusters (k) beforehand. 

Advantage: It produces a dendrogram (a tree-like diagram) that visualizes the hierarchy of merges.  You can inspect this diagram to identify natural cut points and determine the appropriate number of clusters based on the data structure, rather than guessing $k$ or relying on heuristic metrics like the Elbow method.

2. Need for Interpretability and Hierarchy
Choose hierarchical clustering when understanding the relationships between clusters is as important as the clustering itself. 

Advantage: It reveals nested structures (e.g., species within genera, sub-segments within market groups). This is critical in fields like bioinformatics (phylogenetic trees), taxonomy, and social network analysis, where the "parent-child" relationship between groups provides meaningful context that flat K-means partitions cannot.

3. Non-Spherical Cluster Shapes
While K-means assumes clusters are spherical and of similar size, hierarchical clustering (depending on the linkage method used, such as single or complete linkage) can better capture irregular, elongated, or nested cluster shapes. 

Note: It is less constrained by the "spherical" assumption, making it more flexible for complex data distributions, though it can still struggle with varying densities.

### 12. Dimensionality Reduction
Explain the difference between PCA, t-SNE, and UMAP for dimensionality reduction.

Principal Component Analysis (PCA) is a linear technique that projects data onto orthogonal axes to maximize variance, making it fast, deterministic, and highly interpretable for preserving global structure.  In contrast, t-SNE and UMAP are non-linear methods designed to preserve local structure, making them superior for visualizing complex clusters and manifolds but less suitable for downstream machine learning tasks. 

t-SNE focuses almost exclusively on local relationships, making it excellent for detailed 2D visualization of small-to-medium datasets but slow and unable to project new data points.  UMAP offers a balance by preserving both local and some global structure, is significantly faster than t-SNE, scales better to large datasets, and supports projecting new data into existing embeddings.

When would you use each? 
I would use Principal Components analysis if my data was linear and if interpretability was important. We can interpret the results directly from the eigen value/vector decomposition and can map the original feature loading onto the new dimensions to see how they are represented in the transformed space.

I would use t-SNE if visualization was my primary goal and if my data was nonlinear. t-SNE does a great job of preserving the high-dimensional distances between observations on a two dimensional surface.

I would use UMAP if my data was non linear and if I wanted to use the space for feature extraction - keeping the embeddings for new observations in a ML pipeline. In UMAP the local structure is preserved along with some of the global structure as opposed to PCA which preserves the global structure.

How do you choose the number of dimensions to reduce to? 
You decide how much variance within the data you are willing to part with. Usually in PCA, one keeps enough eigenvectors to maintain 80% of the variance, determined by normalizing the eigen values and keeping eigen values until at least 0.8 is accrued.

What information can be lost?
Variability is inevitably lost when the dimensionality is reduced. If you had an orthogonal feature with respect to all of the other features and it had very little variability, then it could (in theory) be lost. All of these dimensions in PCA are orthogonal, so each dropped feature is truly novel information that is being omitted.

## Your Resume Experience: Reinforcement Learning & Optimization

### 13. Reinforcement Learning Basics
Explain the reinforcement learning framework (state, action, reward).
An agent interacts with an environment (state space S, action space A). At each step:
Agent observes state s ∈ S
Takes action a ∈ A
Receives reward r and transitions to next state s'
Goal: Learn a policy π(a|s) that maximizes cumulative discounted rewards: E[Σγᵗrₜ].
Key insight: The Markov property—future only depends on current state, not history.

An MDP is formally defined by a tuple of components:

States ($S$): The possible situations or configurations of the environment. 
Actions ($A$): The set of choices available to the agent. 
Transition Function ($T$): Probabilities of moving from one state to another given an action. 
Reward Function ($R$): Immediate feedback received for taking an action in a state. 
Discount Factor ($\gamma$): A value between 0 and 1 that weighs the importance of future rewards compared to immediate ones.
The primary goal in solving an MDP is to find an optimal policy ($\pi$), which is a strategy mapping states to actions that maximizes the expected cumulative reward over time.

How is it different from supervised learning?
SL	RL
Passive: learn from fixed labeled data	Active: learn via interaction & feedback
No temporal aspect	Temporal: actions affect future states
No exploration problem	Must balance exploration vs exploitation
Evaluate once on test set	Learn continuously from environment

What's the exploration-exploitation tradeoff?
Early in training, your policy is weak because you haven't explored the action space. Two extreme strategies:

Exploit only: Always take best-known action, miss potentially better policies
Explore only: Take random actions, no improvement
Need a balance:

ε-greedy: With probability ε, explore randomly; with 1-ε, exploit best action
UCB (Upper Confidence Bound): Prioritize actions with high uncertainty
Thompson sampling: Sample from posterior over Q-values
Entropy regularization: Add entropy bonus to policy, encourage exploration
Example: In pricing (my domain), ε-greedy means 95% of the time use best price from data, 5% test new prices. This finds locally optimal prices while discovering better ones.

### 15. Demand Forecasting Challenges
What are the unique challenges in demand forecasting vs other prediction tasks? 
Demand forecasting is challenging because it is time dependent where other prediction tasks aren't necessarily time dependent. Time presents unique challenges due to it's temporal relationship to nearby observations. Similarly, our observations aren't random, but typically do depend on previous observations. This makes splitting the data into training and testing necessarily a sequential split as opposed to a random split. It also means that our features have to be robust against any information leakage. We can't include information in the row that wouldn't be known at the time of inference in order to stay consistent with the production data generating process. Environments can arbitrarily shift over time as well. Just because a pattern has been consistent historically, doesn't mean that it always will, so we need to have monitoring tools in place to detect data drift and performance decay.

How do you handle seasonality, trend, and autocorrelation? 
seasonality can be handled in multiple ways. One of the main ways is to take the annual cycle of the targets values over repeated years, de-trend the data (if applicable) then compute the average over the pattern and store this as a seasonality lookup feature for new observations. Trend is different from seasonality, while the target may oscillate depending on time of year, the general movement of the target may be heading in a particular direction (up or down). One way to observe this is to compare the average distance between observations of same-time-last-year data over repeated years. If there is a general consensus movement between them, then you have a trend. autocorrelation is where previous steps are very good predictors of current steps. You can visualize autocorrelation by plotting the target at time T against the target at time T - 1 (or some increment), and if the relationship is highly correlated, then you have autocorrelation. Autocorrelation can be a useful feature in a forecast, but it can only be accessed up to the increment that the autocorrelation was used for. If autocorrelation becomes weak at T - 7, then it's difficult to use autocorrelation as a feature beyond 7 days of the forecast.

What models did you use at TakeUp?
XGBoost with multi-step forecasting (T+1 to T+30). Key decisions:

Multi-step approach: Direct method—separate model for each day (avoids error accumulation from recursive approach).
Features: Lagged values (T-1, T-7, T-365), seasonality indicators, trend, day-of-week, holidays, event flags, price.
Hierarchical forecasting: Day-level and month-level models. Reconcile using optimal approach (ensure month = sum of days). Avoids cumulative error from naive summation.
Why XGBoost: Handles exogenous variables well (price, occupancy), captures feature interactions, faster than ARIMA/LSTM for our latency constraints.
Result: 30-day rolling forecast enabling revenue optimization (my pricing RL system used this as input).

### 16. Online Learning vs Batch Retraining
When would you use online learning (continuous updates) vs batch retraining (daily/weekly)?

I would use online learning in an environment that changes frequently and we need to weight newer observations much higher than older ones. I would also consider online learning in situations where we know little to nothing for newer observations and need to adapt our predictions quickly as we encounter new information about an observation.

I would use batch retraining in an environment that doesn't change as frequently. If I can learn stable relationships of an environment that don't change as often, then I can get away with my learnings over the history and retrain to update those learnings on new data when drift indicators say that the environment has changed enough to warrant it.

What are the tradeoffs in terms of data freshness, computational cost, and stability?

Dimension	Online Learning	Batch Retraining
Data Freshness	Latest observations immediately	Stale (missing recent data)
Serving Latency	High (model updates in real-time, slow to serve)	Low (static model, instant predictions)
Computational Cost	High (continuous gradient updates)	Low (one update per cycle)
Stability	Unstable (gradient noise, catastrophic forgetting)	Stable (validated before deploy)
Validation	Limited (can't A/B test each update)	Comprehensive (evaluate metrics pre-deploy)
Why online learning can be unstable:

Gradient noise: Single examples create noisier updates than batches
Catastrophic forgetting: New data can overwrite old patterns
No validation gate: Bad updates go live immediately
Practical hybrid approach:
Retrain batch daily or weekly, warm-start with previous weights. This captures recent drift while reducing noise.

## Your Resume Experience: Causal Inference

### 17. Causal Inference Fundamentals
Explain the difference between correlation and causation from a causal inference perspective.
Correlation is where two variables share a similar relationship or have a high degree of explained variation. Causation is where we can say diffinitely that the relationship of one variable is effected by a change in the other variable.
What's a confounder? 
A confounder is a variable that causes both the exposure and the outcome (e.g., Age → Smoking and Age → Lung Cancer)
Confounders require control: You must adjust for confounders (via regression, stratification, or matching) to block back-door paths and remove bias. Failing to control for a confounder mixes the exposure effect with the confounder's effect, leading to incorrect estimates.
What's a collider? 
a collider is a variable that is caused by both the exposure and the outcome (e.g., Smoking → Hospitalization ← Lung Cancer)
Colliders must NOT be controlled: Adjusting for a collider opens a "back-door path" between the exposure and outcome, inducing a spurious association (collider bias) even if none exists. This commonly occurs through selection bias (e.g., restricting a study to hospitalized patients) or statistical adjustment for a variable influenced by both the predictor and the result.

### 18. Randomized Controlled Trials (RCTs)
You mentioned causal evaluation pipelines. Walk us through how you'd design an RCT to measure whether a medical intervention actually works.
A very simple RCT design to measure whether a medical intervention actually works is to start by defining the population that you want to study the effect size for this particular medical intervention, then recruit subjects within this population who are willing to participate in a study. Assuming we have a single treatment and aren't interested in dose-responses, we would randomly separate our group into two samples. If there is already an existing treatment standard, then we could assign our fisrt group to this intervention. If there is not already an existing treatment standard, then this group could be assigned a placebo intervention. The idea here is to have a group where all confounding bias is randomly selected away so that we can gather a all-things-being-equal baseline to compare against. Group 2 would be assigned the medical intervention that we are evaluating. Over the study, we would collect our results and build two distributions of the responses for each group. The null hypothesis would be something like Ho: mu1 - mu2 = 0, while ha: mu1 - mu2 != 0 (two-tailed) with type 1 error of 0.05. If the variances are expected to be the same then we can pool them (more statistical power), else we will use a Welch’s t-test (less statistical power, but doesn't assume equal variances). We then calculate our t-test statistic, see where that falls as a p-value in our t distribution given the degrees of freedom minus 1. If the two-tailed p-value is less than 0.05, then we reject the null hypothesis, if the p-value is greater than 0.05, then we fail to reject the null hypothesis.

What's the gold standard and why?
RCTs are the gold standard. Really, being able to mitigate confounding bias through randomization is the gold standard. When our samples aren't random, we allow for unintential bias to conflate the effect that we were hoping was due to the treatment we were testing. Without randomization, it becomes difficult to say whether the result was due to the treatment or if it was due to some latent effect in the way the samples were distributed.

How do you handle data loading from S3?

s3://my-bucket/data/training/   ← training data
s3://my-bucket/data/validation/  ← validation data

When creating a SageMaker training job, I specify these channels:

training_input_config = [{

### 29. Real-Time vs Batch Inference
You need to make predictions for 100M users. Do you use SageMaker real-time endpoints or batch transform? What are the:
cost, 
latency, 
and throughput tradeoffs?

For 100M users, I'd use Batch Transform, not real-time endpoints. Here's my reasoning:

Real-Time Endpoints:
Latency: Sub-100ms response (great for interactive use cases)

Cost: High—instances run 24/7 even with no traffic. ml.m5.xlarge ≈ $0.20/hour = $1,752/month

Throughput: Limited by instance capacity (maybe 100-1000 predictions/min per instance)

Best for: Interactive features (user makes request, gets result immediately)

Batch Transform:
Latency: Hours (job runs once daily or weekly)
Cost: Low—pay only when running. 100M predictions on ml.m5.2xlarge ≈ $50-100/day
Throughput: Massive—processes millions of predictions in parallel
Best for: Precomputed predictions (scores saved to database, served from cache)

For 100M users:
Real-time math:
100M predictions/day ÷ 86,400 seconds ≈ 1,157 predictions/sec
Would need ~10+ ml.m5.xlarge instances ($20k/month)
Always on, wasteful
Batch transform math:

100M predictions × $0.001 per prediction (rough cost) ≈ $100
Run job at 2am when no one is watching
Results written to S3, loaded into Redis/cache for serving
My decision:
Use Batch Transform for the 100M predictions, write results to S3, then populate a real-time cache (Redis/DynamoDB) that serves predictions instantly. Best of both worlds:

Low latency for users (sub-ms from cache)
Low cost (batch job is cheap)
High throughput (process entire user base once daily)
Trade-off: Predictions are slightly stale (updated daily), but that's fine for most use cases (recommendations, personalization, scoring). If you need real-time for a subset (active users), hybrid approach: batch for 95%, real-time endpoint for top 5% high-value users.

### 30. Real-Time Low-Latency Inference Pipeline
You're building a CTR prediction system for real-time bidding. You have a 50-100ms total latency budget, your model takes 22ms to run, and you need to serve 1M requests per second with p99 latency < 20ms.

Walk through:
- How would you architect the end-to-end system (feature fetching, inference, response)?
- Where are the latency bottlenecks and how do you identify them?
- Your model is 22ms but SLO is 20ms—what optimization techniques would you use?
- How do you safely test a new model without breaking the latency SLO?
- How do you monitor to ensure you're continuously meeting latency targets?

**Answer:**

**1. End-to-End System Architecture**

When a bid request arrives, here's the timeline:

```
Bid Request (t=0ms)
  ├─ Feature Lookup (8-12ms) ← Redis/DynamoDB
  │  └─ User history, segment, device info
  ├─ Model Inference (22ms) ← XGBoost prediction
  │  └─ Run through 100 trees, get CTR score
  ├─ Post-processing (1-2ms) ← Business logic
  │  └─ Apply price floor, inventory checks
  └─ Response (t=21-25ms) ✓ Well under 50-100ms budget
```

Current latency breakdown:
- Feature lookup: 10ms (50% of time)
- Inference: 22ms (110% of time) ← OVER SLO!
- Post-processing: 2ms
- **Total: 34ms** (14ms over target)

**2. Identifying Bottlenecks**

I'd measure continuously using latency tracking:

```python
from inference.realtime_optimizer import LatencyTracker

tracker = LatencyTracker(slo_ms=20.0)

# Record every prediction
tracker.record(
    feature_lookup_ms=10.2,
    inference_ms=22.1,
    post_processing_ms=1.5
)

# Analyze patterns
percentiles = tracker.get_percentiles()
print(f"p99: {percentiles['p99_ms']:.1f}ms")  # 34ms

breakdown = tracker.get_breakdown()
print(f"Inference: {breakdown['inference_pct']:.0f}%")  # 65% of time
```

Finding: Inference (22ms) is the bottleneck, not features.

**3. Optimization: 22ms → <20ms**

Multi-pronged approach:

A) **Reduce tree count** (quickest)
   - 100 trees → 75 trees: 22ms → 19.5ms (saves 2.5ms)
   - Accuracy loss: ~1-2%

B) **Optimize feature lookup** (biggest impact)
   - Pre-warm Redis with hot user features
   - Ensure 95%+ cache hit rate
   - Savings: 10ms → 5ms (saves 5ms!)

C) **Model quantization**
   - float64 → float32: saves 0.5ms
   - Accuracy loss: <0.1%

**Result: 22ms + 10ms = 32ms → 19.5ms + 5ms = 24.5ms** ✓ Under SLO!

**4. Safe Testing with Canary Deployment**

Deploy both models simultaneously:

```python
from inference.realtime_endpoint import MultiVariantEndpoint

endpoint.create_multi_variant_endpoint(
    endpoint_name="ctr-ab-test",
    model_configs=[
        {
            "variant_name": "production",
            "model_name": "xgb-100-trees-v1",
            "initial_variant_weight": 0.95,  # 95% traffic
        },
        {
            "variant_name": "canary",
            "model_name": "xgb-75-trees-v2",
            "initial_variant_weight": 0.05,  # 5% traffic
        }
    ]
)
```

Rollout schedule:
- **Hours 0-2**: 95/5 split → monitor p99 latency, CTR accuracy, errors
- **Hours 2-4**: 50/50 split → if metrics look good, continue
- **Hours 4-6**: 10/90 split → final confidence check
- **Hour 6+**: 0/100 split → full rollout (instant if issues)

Why this matters: Catch latency regressions on 5% traffic, not 100%. Rollback is instant (no redeployment).

**5. Continuous Monitoring**

Set up CloudWatch alarms:

```
Alert conditions:
- p99 latency > 20ms → Page on-call
- p95 latency > 18ms → Early warning
- Feature cache hit rate < 95% → Investigate
- Error rate > 0.1% → Investigate

Daily metrics dashboard:
- p50/p95/p99 latencies per variant
- Feature lookup vs inference breakdown
- CTR accuracy comparison
- Canary rollout progress
```

Example: If p99 drifts to 21ms on day 2, either roll back 20% or increase cache size immediately.
